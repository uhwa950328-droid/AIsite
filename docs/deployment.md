# 배포 가이드

이 프로젝트는 **GitHub 저장소와 호스팅(Vercel 등)을 연결**해 두었습니다. 로컬에서 빌드 명령을 따로 실행하지 않아도, **원격 브랜치에 푸시하면 자동으로 빌드·배포**됩니다.

## 배포되는 방식

1. 로컬에서 변경 사항을 커밋합니다.
2. 기본 브랜치(보통 `main`)에 `git push`합니다.
3. 연동된 서비스가 새 커밋을 감지하고 프로덕션 빌드를 실행한 뒤 사이트를 갱신합니다.

배포 로그·성공 여부·프로덕션 URL은 **연동해 둔 호스팅의 대시보드**(예: Vercel → 해당 프로젝트 → Deployments)에서 확인할 수 있습니다.

## 로컬에서 할 일 (푸시 전)

- `npm run build`로 로컬 빌드가 통과하는지 확인하면, 배포 단계에서 실패할 확률을 줄일 수 있습니다.

## 환경 변수

Supabase 연동을 쓰려면 **호스팅 쪽 “Environment Variables”**에 로컬 `.env.local`과 같은 항목을 넣어야 합니다. (값은 저장소에 커밋하지 마세요.)

| 변수명 | 설명 |
|--------|------|
| `NEXT_PUBLIC_SUPABASE_URL` | Supabase 프로젝트 URL |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Supabase anon(공개) 키 — 클라이언트 리뷰 작성·Realtime에 필요 |

서버 전용 조회만 쓰는 경우 일부 배포에서는 `SUPABASE_URL`, `SUPABASE_ANON_KEY` 조합도 코드에서 지원합니다. 브라우저에서 리뷰 입력 등이 필요하면 **`NEXT_PUBLIC_*` 두 변수는 반드시** 설정해야 합니다.

설정 변경 후에는 대시보드에서 **재배포(Redeploy)** 하거나, 빈 커밋을 푸시해 새 빌드를 돌리면 반영됩니다.

## 참고

- 저장소: GitHub 원격과 동기화된 브랜치 기준으로 배포됩니다.
- CLI로 직접 배포하고 싶다면 해당 호스팅 문서(Vercel CLI 등)를 따르면 됩니다. 일상적인 플로우는 **Git 푸시 한 번**으로 충분합니다.
